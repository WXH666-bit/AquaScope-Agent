# 会话记录与执行轨迹存储说明

## 1. 界面里的“轨迹”是什么

界面显示的是可审计的执行轨迹，包括 LangGraph 节点、路由结果、检索器调用、
MCP 工具调用、证据数量、自动评估与重试结果。它不是、也不会保存模型私有思维链。

## 2. 文件与数据库位置

| 内容 | 保存位置 |
| --- | --- |
| 会话、消息、附件、证据、执行轨迹 | `data/mrag/sessions/web_app.sqlite` |
| LangGraph checkpoint | `data/mrag/sessions/langgraph.sqlite` |
| 每个会话的长期记忆 JSON | `data/mrag/sessions/{session_id}.json` |
| 用户上传附件 | `data/mrag/uploads/{session_id}/` |
| 网络图片缓存 | `data/mrag/network_images/` |

`web_app.sqlite` 中的关键表：

- `sessions`：会话标题、收藏状态和更新时间。
- `messages`：用户与助手消息；同一个 `turn_id` 表示一轮对话。
- `attachments`：上传图片、PDF 的路径和元数据。
- `evidence`：Chroma、LightRAG、PDF 图片、Wikipedia 等证据。
- `trace_events`：节点、工具和评估的执行记录。

## 3. 为什么侧栏显示“轮对话”

一轮对话包含一条用户消息和一条助手消息，所以数据库中通常是两条 message。
侧栏使用 `COUNT(DISTINCT turn_id)` 显示“1 轮对话”，不会再把它误写成“2 条消息”。

## 4. 图片返回规则

每次回答最多展示一张图片，优先级为：

1. 当前问题命中的 PDF 原始提取图片；
2. 项目知识库中保存的本地图片；
3. Wikimedia Commons 网络图片。

当用户询问“它的天敌并给我图片”时，图片目标是具体天敌，而不是再次返回原生物。
系统先调用大模型生成联网检索计划，再使用 Wikipedia API 获取文本证据，并从
Wikimedia Commons 获取一张可追溯来源的图片。
