# AquaBio 当前版本运行与调用流程

## 1. 先分清两个版本

当前仓库包含两个可独立运行的检索子系统；最终双索引融合尚未接入主 LangGraph。

| 版本 | 状态 | 存储 | 能否运行 |
|---|---|---|---|
| AquaBio-MRAG | 已实现 | BGE-M3 + Chroma | 可以 |
| RAG-Anything PDF 子系统 | 已实现并完成小段验收 | LightRAG + NetworkX + NanoVectorDB | 可以 |
| Chroma + LightRAG 最终融合 | 待接入主 LangGraph | Weighted RRF | 尚未接入 |

当前真正的主程序：

```text
mrag_cli.py
src/aquabio_mrag/
data/mrag/vector_db/chroma/
```

当前已经存在：

```text
raganything_cli.py
src/aquabio_raganything/
data/mrag/raganything/working/
.venv-raganything/
```

因此以下命令现在可以运行：

```cmd
raganything_cli.py inventory
raganything_cli.py index --scope relevant
raganything_cli.py query --mode hybrid ...
```

详细命令、真实存储数量和 PDF 操作见项目根目录
`学习rag anything的构建`。

## 3. 当前在线问答文件调用流程

以命令为例：

```cmd
.\.venv\Scripts\python.exe mrag_cli.py ask --query "海星有哪些外观特征？"
```

真实调用顺序：

```text
mrag_cli.py
  |
  +-- MRAGPaths.from_root()
  |     └── src/aquabio_mrag/config.py
  |
  +-- MRAGSettings.from_env()
  |     ├── .env
  |     └── src/aquabio/config.py::load_env()
  |
  +-- AquaBioMRAGWorkflow()
        └── src/aquabio_mrag/workflow.py
              |
              +-- MultiSourceRetriever()
              |     └── src/aquabio_mrag/retrieval.py
              |           |
              |           +-- ChromaMRAGStore()
              |                 └── src/aquabio_mrag/vector_db.py
              |                       |
              |                       +-- BGEEmbedder
              |                       |     └── F:\huggingface\aquabio
              |                       |
              |                       └── Chroma PersistentClient
              |                             └── data/mrag/vector_db/chroma
              |
              +-- OpenRouterClient()
              |     └── src/aquabio/openrouter.py
              |
              └-- LangGraph.invoke()
```

LangGraph 节点顺序：

```text
START
  ↓
session_init
  ↓
memory_load
  ↓
followup_resolver
  ├── 无历史指代 → human clarification → memory_save → END
  └── 可继续
        ↓
router
  ↓
rewrite
  ↓
source_selection
  ↓
react_tool_plan
  ↓
[有图片时 vision]
  ↓
retrieval_agent 子图：retrieval → rerank → context
  ↓
answer_agent 子图：answer → response_guard → evaluation
  ├── 不通过 → retry → retrieval/rewrite/vision/answer_agent
  └── 通过
        ↓
     finalize
        ↓
     memory_save
        ↓
       END
```

### 3.1 Router

`workflow.py::router_node()` 根据输入识别：

```text
text_qa
image_qa
multimodal_qa
comparison_qa
source_trace
pdf_qa
```

### 3.2 Rewrite

给原问题补充领域检索词。例如比较问题会强调：

```text
视觉特征
身体结构
相似物种
image_recognition_tips
权威 PDF 描述
```

### 3.3 Source Selection

选择当前需要的工具：

```text
text_retriever
image_retriever
multimodal_retriever
pdf_retriever
vlm_caption
```

### 3.4 Retrieval

问题先用 BGE-M3 生成 1024 维查询向量，再去 Chroma 做余弦相似度查询。

检索后不是直接使用相似度，而是二次重排：

```text
final_score =
  0.45 * semantic_similarity
  + 0.20 * species_match
  + 0.15 * source_weight
  + 0.10 * chunk_weight
  + 0.10 * keyword_overlap
```

### 3.5 Context

将文本、图片、图文配对和 PDF 结果去重，最多组织 24 条：

```text
[E1] ...
[E2] ...
```

### 3.6 Answer

OpenRouter 只允许根据检索证据回答，关键结论必须引用 `[E数字]`。

### 3.7 Evaluation

检查：

- 是否检索到上下文。
- 是否有完整引用。
- 答案是否被截断。
- 图片任务是否有 caption。

不通过时最多重试 2 次。

## 5. 当前 PDF 究竟做了什么

当前 PDF 流程是 PyMuPDF 文本 RAG，不是 RAG-Anything。

### 5.1 登记和下载

`configs/pdf_sources.json` 保存：

```text
doc_id
title
source_org
doc_type
local_path
source_url
usage
priority
```

下载时：

1. 检查本地文件是否以 `%PDF` 开头。
2. 不存在时下载。
3. 校验 HTTP 状态。
4. 保存文件大小和 SHA-256。
5. 写入 `pdf_registry.jsonl`。

### 5.2 文本提取

`pdf_pipeline.py::parse_registered_pdfs()`：

1. 用 PyMuPDF 逐页打开。
2. `page.get_text("text")` 提取文本。
3. 合并英文断词。
4. 清理空格和多余换行。
5. 每块约 1800 字符。
6. 相邻块重叠约 250 字符。
7. 保存 PDF 标题、机构、页码、章节和来源 URL。
8. 根据物种英文名、学名和关键词标记 `related_species`。

当前产生 192 个 `pdf_chunk`。

### 5.3 PDF 图片

脚本支持：

```cmd
.\.venv\Scripts\python.exe scripts\04_download_parse_pdfs.py --extract-figures
```

但当前图片处理很基础：

1. `page.get_images()` 找到 PDF 内嵌图像。
2. `document.extract_image()` 保存图像。
3. 取该页前 600 字符作为 nearby text。
4. 根据 nearby text 关联物种。
5. 将“来源文档 + 页码 + nearby text”作为图片文本描述。

当前实际清单中的 `pdf_figures=0`，因为正式构建时没有启用 `--extract-figures`。

当前 PDF 图片没有：

- VLM 视觉分析。
- 表格结构恢复。
- 公式处理。
- 实体关系抽取。
- 图结构存储。

这些属于计划中的 RAG-Anything 阶段。

## 6. 当前外部图片究竟做了什么

20 类物种，每类 10 张，共 200 张。

来源：

```text
Wikimedia Commons
iNaturalist
```

每张图片：

1. 下载到 `data/mrag/images/{species_id}/`。
2. 用 Pillow 验证能否打开。
3. 检查宽高不小于 256。
4. 保存来源页、作者、许可证和原图 URL。
5. 保存 Commons/iNaturalist 已有描述作为 caption。
6. 拼接中英文名、学名、关键词和 caption，得到 `embedding_text`。

图片本身没有直接送入 BGE-M3。BGE-M3 是文本模型，实际过程是：

```text
图片
  ↓
已有 caption + 物种关键词
  ↓
embedding_text
  ↓
BGE-M3 文本向量
  ↓
Chroma
```

系统还给每张图片匹配同物种的：

```text
overview
visual_features
habitat
```

生成 200 条 `multimodal_pair`：

```text
图片 caption + 同物种文本
  ↓
图文配对 embedding_text
  ↓
BGE-M3 + Chroma
```

因此当前所谓“多模态向量库”准确地说是：

```text
caption-based multimodal RAG
```

不是 CLIP 原生图像向量，也不是图像像素直接向量化。

在线用户上传图片时流程不同：

```text
用户图片
  ↓
OpenRouter VLM
  ↓
description + visible_features + possible_species
  ↓
用生成的文字查询 Chroma
```

这仍然是“图片转文字后检索”。

## 7. 当前有没有图结构

没有。

当前 `src/aquabio_mrag/` 中没有：

```text
Neo4j
NetworkX
LightRAG
entity graph
relation graph
```

当前关系只通过 JSON 元数据表达，例如：

```json
{
  "image_id": "img_starfish_001",
  "text_ids": [
    "starfish_overview",
    "starfish_visual_features",
    "starfish_habitat"
  ]
}
```

这叫显式 ID 配对，不是可遍历知识图谱。

## 8. 为什么设计里不用 Neo4j

RAG-Anything 的图结构必须有图存储，但图存储不等于必须使用 Neo4j。

计划首版使用：

```text
NetworkXStorage
```

原因：

- 两份 PDF 的首期范围只覆盖 20 类相关章节。
- 单机课程/原型项目不需要独立数据库服务。
- NetworkX/GraphML 文件便于调试和查看。
- RAG-Anything/LightRAG 原生支持图存储抽象。
- 部署成本低，Windows 本地运行简单。

Neo4j 更适合：

- 百万级节点或关系。
- 多用户并发图查询。
- 复杂 Cypher 查询。
- 图谱需要独立服务和权限管理。
- 多台应用共享同一图数据库。

本项目首版还没有这些需求。

## 9. 原来是否使用过 Neo4j

没有。

仓库现有代码从未建立 Neo4j 驱动、连接配置或数据库。之前的版本经历了：

```text
旧原型：TF-IDF 稀疏向量
  ↓
当前主版本：BGE-M3 + Chroma
  ↓
下一阶段设计：Chroma + RAG-Anything/LightRAG
                    └── NetworkXStorage
```

所以不是“把 Neo4j 改成 NetworkX”，而是：

```text
以前根本没有图数据库
现在计划第一次加入图结构
首版选择 NetworkX，而不是 Neo4j
```

## 10. 下一阶段 RAG-Anything 会对 PDF 和图片做什么

计划中的真实变化：

```text
PDF
  ↓
MinerU 分解为 text/image/table/equation
  ↓
RAGAnything.insert_content_list()
  ├── 文本 → LightRAG chunk → Entity/Relation
  ├── 图片 → ImageModalProcessor → VLM 描述 → 图片实体
  ├── 表格 → TableModalProcessor → 结构与摘要 → 表格实体
  └── 公式 → EquationModalProcessor → 公式解释 → 公式实体
  ↓
LightRAG
  ├── JsonKVStorage
  ├── NanoVectorDBStorage
  ├── NetworkXStorage
  └── JsonDocStatusStorage
```

图中可以出现：

```text
sea_urchin --has_feature--> spines
sea_urchin --is_a--> Echinoidea
image_001 --belongs_to--> sea_urchin_figure
sea_urchin_figure --illustrates--> sea_urchin
table_001 --lists--> Echinoidea_taxonomy
text_001 --describes--> sea_urchin_habitat
```

检索时：

```text
LightRAG mode="hybrid"
  =
向量语义检索
+
实体/关系图检索
+
跨模态关系检索
```

然后与当前 Chroma 结果使用 Weighted RRF 融合：

```text
score =
0.55 / (60 + lightrag_rank)
+
0.45 / (60 + chroma_rank)
```

RAG-Anything 子系统已实现并能返回图谱实体、关系和页码证据；主
`src/aquabio_mrag/workflow.py` 尚未并行调用该 MCP，因此原 `mrag_cli.py`
轨迹暂时不会显示 `graph_context`。
